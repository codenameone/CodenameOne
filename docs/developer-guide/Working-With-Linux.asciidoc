== Working with the native Linux port

The native Linux port compiles a Codename One app to a real, standalone Linux
executable -- a single ELF binary that runs with *no JVM*. It's the Linux analog
of the iOS and native Windows ports: the same ParparVM pipeline that turns your
Java/Kotlin bytecode into C and then into a native binary, here targeting Linux
and rendering through GTK3 / Cairo / Pango.

This is distinct from the long-standing *Linux desktop* build, which packages the
app to run on a Java Virtual Machine (see
<<_native_linux_vs_the_desktop_jvm_app_vs_the_jar>> for a practical comparison).
The native port needs no JVM on the target machine.

=== How it works (the technology stack)

The port reuses Codename One's portable architecture and swaps in a GTK3 / Cairo
implementation layer:

- *ParparVM "clean" C target.* The same VM that powers iOS and Windows translates
  your app + the Codename One core + the minimal Java runtime to C. A CMake project
  is generated and compiled with `cc`/Clang + *Ninja* into a native ELF. There is
  no bytecode interpreter and no JNI -- your code *is* the native binary.
- *Cairo + Pango + GdkPixbuf.* All 2D graphics (primitives, gradients, clipping,
  affine transforms, images) go through Cairo; glyph layout, measurement and
  rasterization use Pango (with FontConfig/FreeType, so bundled fonts -- including
  the material icon font -- register and render); image decode/encode uses
  GdkPixbuf. Windowing and native peers (text editing, browser, video) are GTK3
  widgets layered over the Cairo drawing surface.
- *OpenGL ES (EGL)* backs the portable 3D API (`com.codename1.gpu`); *GStreamer*
  backs `Media` playback, audio recording and the camera; *WebKitGTK* backs
  `BrowserComponent`; *libsecret* backs secure storage; *libnotify* backs local
  notifications; *GeoClue* backs location; *libcurl* and POSIX sockets back
  networking.
- *Single self-contained executable.* There is no `.app`-style bundle directory,
  so the app's classpath resources -- the theme `.res`, images, localization, the
  material icon font -- are embedded directly into the ELF and read back at
  runtime through `getResourceAsStream`. The result is one file you copy and run.

=== Why GTK3 (and not GTK4)

The port targets *GTK3*. GTK4 isn't a drop-in successor -- the two are
parallel-installable by design because they're source-incompatible and
ABI-incompatible (different sonames, renamed and removed APIs), so an app links
one or the other, not both. GTK3 was chosen for *reach*: it has shipped on every Linux
desktop since 2011 (the GTK3 runtime is present, directly or transitively, on
virtually any machine with a graphical session), whereas GTK4 (2020 onward) is
common on recent GNOME systems but not universal -- older LTS releases, lighter
desktops and many enterprise/embedded images still carry only GTK3. Linking GTK3
maximizes the set of machines your binary runs on out of the box. (GTK is
dynamically linked, so a future GTK4 build would be a separate target rather than
a flag, to revisit once GTK4's footprint is as universal as GTK3's.)

=== Why the binary links musl, not glibc

The translated runtime and your app's C are built against *musl* libc, statically
linked into the binary, while the GTK stack is dynamically linked. That split is
deliberate, and it's what makes the executable portable.

A glibc-linked binary is tied to the *exact glibc version of the machine that
built it*: glibc versions its symbols, so a binary built on a current distro
fails to even start on an older one with the familiar `version 'GLIBC_2.34' not
found` error. That makes the libc -- the one library every program depends on --
the single most fragile external dependency a native Linux binary has. musl
sidesteps this entirely: it's small and designed to be statically linked, so the
C runtime travels *inside* the executable. There is no glibc version to match, no
per-distribution surprise -- the binary just runs.

The GTK libraries themselves stay dynamically linked (the GTK stack is large and
not designed to be statically linked, and every desktop already ships it with a
stable ABI), so the binary resolves `libgtk-3`, Cairo, Pango, etc. from the system
at startup. The net effect: the fragile part (libc) is baked in, and the part
that's genuinely a platform service (the desktop toolkit) is shared -- a single
small ELF that "just works" across distributions.

=== Building a native Linux app

A native Linux build runs through the `codenameone-maven-plugin` like every other
target. Build it with:

[source,bash]
----
mvn -pl common package -Dcodename1.platform=linux -Dcodename1.buildTarget=local-linux-device cn1:build
----

The builder translates the app to C with ParparVM's `linux` app type, generates a
CMake project that links the GTK stack via `pkg-config`, builds it with CMake +
Ninja, and produces the ELF.

==== Build-host dependencies

The native compile needs the GTK development stack and the capability libraries
the port links. On Debian/Ubuntu:

[source,bash]
----
sudo apt-get install -y \
  cmake ninja-build pkg-config \
  libgtk-3-dev libcairo2-dev libpango1.0-dev libgdk-pixbuf-2.0-dev libglib2.0-dev \
  libfontconfig1-dev libfreetype-dev libcurl4-openssl-dev \
  libgstreamer1.0-dev libgstreamer-plugins-base1.0-dev \
  libwebkit2gtk-4.1-dev libsecret-1-dev libnotify-dev libgeoclue-2-dev \
  libepoxy-dev libegl1-mesa-dev libgles2-mesa-dev
----

The `*-dev` packages are needed on the *build* host. The machine that *runs* the
binary only needs the corresponding runtime libraries, which any standard desktop
already has.

==== Architecture and cross-compiling

Both *x64* and *arm64* are supported; pick the target with the `linux.arch` build
hint. The build is a *native* compile -- it produces a binary for the host
architecture by default, so the straightforward way to get both is to build on a
machine of each.

You can also *cross-compile* -- and you may well want to, for example producing an
arm64 binary from an x64 build host. The translated C is architecture-independent,
and the toolchain selected for the musl link (`zig cc`) is itself a
cross-compiler, so it can emit a binary for a different CPU from the one it runs
on. Point `linux.cc` at a cross `zig cc` wrapper for the target triple (or set
`linux.toolchain=zig` together with `linux.arch`) to cross-build.

=== Build hints

The native Linux port adds the build hints below (all use the `linux.` prefix).

.Native Linux port build hints
|===
|Name |Description

|linux.arch
|Target CPU architecture for the ELF: `x64` (default) or `arm64`. The synonyms
`x86_64`/`amd64` and `aarch64` are accepted.

|linux.debug
|`true`/`false` (default `false`). When `false`, the ELF is optimized and
*stripped* (no debug info). Set `true` to keep symbols (`RelWithDebInfo`) so a
native crash address can be turned back into a function name and source line while
developing. Optimizations stay on either way.

|linux.toolchain
|Set to `zig` to build with `zig cc` -- a self-contained cross-compiler that ships
the musl sysroot -- so a glibc host can emit the musl binary (and cross-compile to
a different `linux.arch`).

|linux.cc
|Override the C compiler CMake uses (for example a `zig cc` wrapper for a specific
target triple, a `musl-gcc`, or a cross `cc`). Takes precedence over
`linux.toolchain`; also settable through the `CN1_CC` environment variable.
|===

The full cross-platform build-hints reference is in the
link:#_sending_arguments_to_the_build_server[Advanced topics chapter].

=== Optimized and stripped by default

The shipping build is optimized and stripped: no debug information is embedded, so
the single self-contained executable stays as small as the translated code allows.
Setting `linux.debug=true` keeps the symbols so a faulting address can be
symbolized during development; the build stays optimized either way.

=== Native Linux vs. the desktop (JVM) app vs. the jar

|Yes (any installed JRE)

|What ships
|One small ELF + the system GTK libraries
|App + a packaged JRE
|A single `.jar`

|Rendering
|GTK3 / Cairo / Pango (native)
|Java2D on the JVM
|Java2D on the JVM

|Startup / footprint
|Native, small
|JVM startup, large
|JVM startup

|How it's built

The native port is the right choice when you want a true native Linux binary with
no JVM dependency; the desktop build remains the convenient path for development
and for environments where shipping a JRE is acceptable.

=== Blocking native calls and the concurrent GC

ParparVM's garbage collector is concurrent: before it can traverse a lightweight
(CN1-created) thread's stack it sets that thread's `threadBlockedByGC` flag and
spins until the thread parks itself (clears `threadActive`). A native that parks a
CN1 thread in a long blocking syscall -- a socket `read`/`write`/`connect`, an
`curl_easy_perform`, a `sleep` -- *must* drop `threadActive` across the call
(`CN1_YIELD_THREAD` / `CN1_RESUME_THREAD`), exactly as every other port does.
Forget it on even one path (for example the cn1ss screenshot reader parked in
`read()` waiting on the server) and a GC that happens to fire during that window
never completes, deadlocking every thread waiting on it. The Linux port wraps all
of its blocking I/O this way; keep that invariant when adding new native calls.
