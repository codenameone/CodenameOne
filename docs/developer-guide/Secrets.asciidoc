== Secrets

`com.codename1.secrets.Secrets` is a simple, secure-by-default key/value store for the sensitive strings an app must keep -- auth tokens, refresh tokens, API keys, anything you must never write to `Preferences` or `Storage` in the clear. It's available on **every** Codename One tier, including the free tier.

[source,java]
----
Secrets.set("auth.token", token);
String token = Secrets.get("auth.token");          // null if absent
String token = Secrets.get("auth.token", "");      // with a default
if (Secrets.contains("auth.token")) { ... }
Secrets.delete("auth.token");
java.util.List<String> keys = Secrets.keys();      // names only, never values
----

=== Secure by default

`Secrets` never stores a value as plaintext.

- **On a device with a platform keychain wired in**, values are protected by the OS secure enclave -- **iOS Keychain Services** and **Android Keystore**. This is the recommended production path; `Secrets.isHardwareBacked()` returns `true`.
- **Otherwise** (notably the simulator), `Secrets` falls back to a software store that encrypts each value with authenticated **AES-256-GCM** (a fresh 12-byte nonce per value, tamper detected on read) before writing it to `Storage`. The 256-bit master key is generated once with the platform CSPRNG and kept in `Preferences`. `isHardwareBacked()` returns `false`.

Either way your code is the same four calls -- the storage backend is chosen for you.

==== Choosing based on sensitivity

[source,java]
----
if (Secrets.isHardwareBacked()) {
    Secrets.set("biometric.key", verySensitiveValue);
} else {
    // software-encrypted at rest, but the key lives in app storage --
    // decide whether that is acceptable for this particular value.
}
----

=== Pluggable backend (`SecretsStore`)

The store behind `Secrets` is an SPI, `com.codename1.secrets.SecretsStore`. A platform port or a keychain cn1lib registers a hardware-backed implementation during initialization:

[source,java]
----
Secrets.setStore(new MyKeychainSecretsStore());  // implements SecretsStore
----

A `SecretsStore` implements `set` / `get` / `contains` / `delete` / `keys` / `isHardwareBacked` over opaque UTF-8 strings, and must never log or transmit plaintext. When none is registered, the AES-at-rest `DefaultSecretsStore` is used.

=== Don't roll your own

Do **not** keep secrets in `Preferences` or `Storage` directly -- both are plaintext on disk. `Secrets` exists precisely so you don't have to think about encryption, key management, or the per-platform keychain APIs.

=== Cloud secrets vault (server-side, for your build credentials)

Aside from the on-device `Secrets` store, Codename One Cloud provides a **server-side secrets vault** for the credentials your *build/back end* needs -- for example the App Store Connect API key and Google Play service-account JSON used by Commerce's server-side validation (see the _Commerce_ chapter). These are managed from the **Secrets** page of the console, are encrypted at rest (AES-256-GCM) with a key held only in the deployment environment, are write-only from the console (a value is never shown again), and are available on **all tiers**. The on-device `Secrets` API and the cloud vault solve different problems: the former protects secrets on the user's device; the latter holds your own server credentials.
