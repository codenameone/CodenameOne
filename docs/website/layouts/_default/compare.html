Framework comparison

The framework choice is a UI ownership choice.

All four approaches can ship useful products. The difference is who controls the UI implementation, how many codebases you maintain, and which platform constraints you accept.

Codename One

One Java codebase, native output, and an app-owned UI.

  • Share application logic and UI across every supported target.
  • Call platform SDKs through native interfaces, including synchronous calls where the platform permits them.
  • Embed maps, browsers, cameras, and other platform views with PeerComponent.
  • Build iOS in the cloud without maintaining a Mac build machine.

Platform UI

Direct platform APIs with separate Swift and Kotlin implementations.

  • UIKit, SwiftUI, Android Views, and Compose expose vendor APIs directly.
  • Each platform has its own UI implementation, tests, and release path.
  • New OS components arrive first in the vendor toolkits.
  • OS updates can change the appearance or behavior of existing components.

Flutter

A larger framework-specific ecosystem built around Dart.

  • Flutter has a large package catalog, mature tooling, and a broad framework community.
  • It owns its pixels through the Flutter rendering engine, as Codename One does through its renderer.
  • Dart is the primary application language; Java has the larger general-purpose ecosystem and hiring pool.
  • Platform channels call native SDKs, while Platform Views embed native controls.

Web

A URL-first deployment model backed by the browser.

  • Instant deployment matters more than app-store distribution.
  • The browser APIs cover the required device integration.
  • Your team and product are web-first.
  • You accept browser-engine behavior and wrapper tradeoffs on app stores.

The Codename One model

The renderer gives the application control of its UI.

Versioned UI

The UI runtime and theme ship inside the application. Shared screens keep one implementation, and an OS update cannot silently replace the component implementation after release.

Current platform styling

Codename One is actively improving fidelity with Liquid Glass and Material 3 themes. You adopt those changes by updating and testing the application rather than waiting for an OS update to restyle it.

Direct native integration

Native interfaces expose platform SDKs without forcing every call through an asynchronous channel. PeerComponent places heavyweight platform views inside the shared UI.
